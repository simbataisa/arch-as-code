### Per-Doc Definition of Done (15 sections)

Every authored doc must contain all 15 sections from the pattern doc template:

1. **Problem Statement** — the specific challenge this pattern addresses in a banking context
2. **Context** — when and where this pattern applies (service tiers, team, deployment context)
3. **Solution** — narrative description + Mermaid architecture/sequence diagram
4. **Implementation Guidelines** — Java 21/Spring Boot 3.x code examples, config snippets
5. **When to Use** — explicit list of qualifying conditions
6. **When Not to Use** — explicit list of disqualifying conditions
7. **Variants** — 2–3 named variants with trade-offs
8. **NFR Acceptance Criteria** — at least one measurable threshold per relevant NFR (latency, availability, RTO)
9. **Compliance Mapping** — 3-ring table; Ring 2 row ends with `⚠️ (working summary — pending Legal review)`
10. **Cost / FinOps** — estimated resource footprint, cost drivers, optimisation levers
11. **Threat Model** — at least 2 named threats (STRIDE category) with specific mitigations
12. **Runbook Stub** — at least one named alert with p50/p99 thresholds and one named remediation step
13. **Test Strategy Stub** — unit, integration, and chaos test outlines
14. **Related Patterns** — cross-links to at least 2 catalog entries
15. **References** — external standards, RFCs, or vendor docs

### Tech Stack Context

All implementation examples use:
- Java 21 / Spring Boot 3.x, Resilience4j, Spring Kafka, Apache Camel
- PostgreSQL 16, Debezium, OPA/Rego, HashiCorp Vault
- React 18 + TypeScript 5 (frontend), Swift 5.10 / Android Kotlin 2.x (mobile)

### Per-Group Gate

After each domain group is authored, run the following before promoting Proposed → Draft:

1. `bash scripts/mermaid-lint-doc.sh <file>` for every file in the group — all blocks must pass
2. `python3 scripts/check-compliance-rows.py` — must report 0 failures
3. Dead-link scan across all newly authored files
4. Update catalog status Proposed → Draft for all docs in the group
5. Commit with message: `feat(catalog): Wave 5X gate — promote <IDs> Proposed→Draft`

| 6F | TPL-001–004, COMP-001 | 5 |

> SEC-006–013 and all other Wave 5 outputs are eligible for a follow-on Wave 6 sweep after the initial 64 are promoted.

### Hybrid Gate (two stages per group)

**Stage 1 — Automated checks** (prerequisite; no doc proceeds if any check fails):

- `bash scripts/mermaid-lint-doc.sh <file>` — 0 failures
- `python3 scripts/check-compliance-rows.py` — 0 failures
- Dead-link scan — 0 broken links in the group

**Stage 2 — Self-review checklist** (per doc; all items must be checked before promotion):

- [ ] All 15 sections present and non-empty (no "TBD", no placeholder text)
- [ ] Mermaid diagram accurately represents the described solution (no stale/mismatched diagram)
- [ ] NFR Acceptance Criteria contains at least one measurable threshold (p99 latency, RTO, availability %)
- [ ] Compliance Mapping covers all 3 rings; Ring 2 row ends with `⚠️ (working summary — pending Legal review)`
- [ ] Threat Model names at least 2 specific threats (STRIDE category) with concrete mitigations
- [ ] Runbook stub has at least one named alert with threshold values and one named remediation step
- [ ] Related Patterns links are valid and bidirectional (the linked doc also references back)

Any doc failing one or more checklist items receives a targeted fix in the same task before the group is promoted. No doc moves to Approved with open items.

### Promotion Commit Pattern

```
feat(catalog): Wave 6X gate — promote <IDs> Draft→Approved
```
