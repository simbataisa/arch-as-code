# Compliance Mapping Matrix — Pattern × Regulation

Status: Draft | Last Reviewed: 2026-05-09 | Owner: @head-of-compliance
Catalog ID: COMP-001 | **Spine**
Tier Applicability: N/A (master cross-reference matrix)

## Problem Statement

Without a single matrix, regulators auditing Techcombank's controls would have to walk every pattern doc to verify that PCI-DSS §3.5 is actually implemented somewhere; engineers would have to re-derive the mapping each time. This master matrix is the **regulatory entry point** to the catalog: pick a regulation, find all patterns that satisfy it. It is sourced from each pattern's Compliance Mapping table and lives next to the pattern docs (not buried in compliance archives) so that engineers and reviewers see it daily.

## Context

Reach for this doc whenever:

- Regulator asks "show me how Techcombank satisfies PCI-DSS §3.5".
- Auditor asks "what is your evidence for BCBS 239 Principle 6?".
- Designing a new pattern — check what regulations adjacent patterns already cover.
- Reviewing a DAB submission — ensure each cited compliance ring has at least one supporting pattern.

## Solution

Authoritative cross-reference table generated from `_compliance-matrix.yml`. Each row is a catalog pattern; each column is a regulation. Cells contain section / control references or `—` (not applicable) or `TBD` (pattern is a stub awaiting authoring).

## Implementation Guidelines

### How a pattern author populates compliance cells

Within a pattern doc, the `## Compliance Mapping` section is structured as:

```markdown
## Compliance Mapping
| Layer | Reference | Section/Control | How this pattern satisfies |
|---|---|---|---|
| Ring 0 (generic) | <regulation-id from _compliance-matrix.yml regulations> | §… | one sentence |
| Ring 1 (intl banking) | <regulation-id> | §… | one sentence |
| Ring 2 (Vietnam) | <regulation-id> | §… | one sentence — flag ⚠️ (working summary) if Vietnamese source |
```

A nightly cron / pre-merge CI job runs `scripts/sync-compliance-matrix.py`, which:
1. Reads every pattern's `## Compliance Mapping` section.
2. Updates the corresponding row in `_compliance-matrix.yml`.
3. Re-runs `scripts/render-compliance-matrix.py` to regenerate the tables in this file.
4. Fails the build if any pattern's mapping table is malformed.

### Render script (Phase 4 enhancement)

```python
# scripts/render-compliance-matrix.py — outline
import yaml
from pathlib import Path

def render_table(mapping: dict, regs: list[dict]) -> str:
    lines = ["| Catalog ID | " + " | ".join(r["name"] for r in regs) + " |"]
    lines.append("|" + "---|" * (len(regs) + 1))
    for cat_id in sorted(mapping):
        row = mapping[cat_id]
        cells = [row.get(r["id"], "—") for r in regs]
        lines.append(f"| {cat_id} | " + " | ".join(cells) + " |")
    return "\n".join(lines)
```

### Lint rules

- **Every Approved-status pattern must have ≥ 1 non-empty cell** across all rings (otherwise the pattern has no regulatory grounding — flag for review).
- **No Vietnamese-source cell may omit the ⚠️ working summary flag** until `@legal-vietnam` sign-off — see `_research-notes.md` for the authoritative working summaries.
- **A regulation column with no cells** is a gap — surface in [§5 Gap Analysis](../../governance/standards/enterprise-architecture-catalog.md#5-gap-analysis) of the catalog.

## Variants & Trade-offs

| Variant | When | Trade-off |
|---|---|---|
| **Single matrix** (this doc) | Default | One file scales to ~150 patterns × 25 regulations |
| **Per-regulation views** (one file per regulation) | Generated for auditor packs | Easier to consume per-audit; same source |
| **Domain-scoped views** (per banking flow) | Generated for domain teams | Custom audience; same source |

## Cost / FinOps Notes

Building and maintaining the matrix consumes ~0.5 FTE-day per quarter (review cycle). The cost of NOT having it: every regulator engagement re-derives the mapping from scratch (often weeks of effort across the org).

## Threat Model Summary

STRIDE: this matrix addresses **Repudiation** of compliance commitments — every claim is traced to a specific control.

- Top threats addressed:
  - Mapping drift between code and compliance — synced via the nightly job.
  - Stale Vietnamese citations — `⚠️ (working summary — pending Legal review)` flag forces Legal review before Approved status.
- Residual:
  - Regulator may interpret a control differently than our cell text — mitigated by including the actual control text from `_research-notes.md`.

## Operational Runbook (stub)

- **Quarterly review**: Compliance team walks the matrix, samples 20 cells, verifies pattern docs match.
- **Regulator-driven update**: when a regulation changes (e.g., PCI-DSS 4.0 → 4.1 or new SBV circular), all affected cells flagged for review and a tracking issue opened.
- **Stub backfill**: as each stub goes to Approved, run `scripts/sync-compliance-matrix.py`.

## Test Strategy (stub)

- **Unit**: render-script tests with fixture YAML.
- **Integration**: round-trip — pattern doc → YAML → matrix; verify cells consistent.
- **Periodic audit**: annual external audit samples 10 cells; verify pattern doc contains the cited control reference.
